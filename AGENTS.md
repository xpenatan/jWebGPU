1. **Analyze**: Understand the user request and existing codebase. **NEVER guess code structures or property names.** You must verify the existence of classes, methods, and fields by reading the source files. (e.g., Do not assume a variable or a method exists; check the class structure first).
2. **Sync State**: Read `LOCAL_AGENT.md` to understand where the previous session left off.

4. **After planning:**: Always ask the user if can execute the plan. User may ask questions about the plan. If user approves, proceed to execute. If user asks for changes, update the plan and repeat step 3.
5. **Execute**: Perform file modifications, terminal commands, etc.
6. **Update Progress**: After a task (or significant sub-task) is completed or verified, update the "Current Progress" and "Next Task" sections in `LOCAL_AGENT.md`.
7. **Verify**: Always run compilation to ensure the state is stable before ending the turn.

This ensures that if the session is interrupted, the next agent has a perfect "snapshot" of the state machine, including why decisions were made and which files were touched.

## Big Picture
- jWebGPU is a generated binding stack: `webgpu-base` (hand-authored templates) + `webgpu-build` (generation/compile driver) -> generated target modules (`webgpu-core`, `webgpu-desktop-ffm`, `webgpu-teavm`) + packaged native artifacts (`webgpu-desktop-jni`, `webgpu-android`).
- Generation entrypoint is `webgpu/webgpu-build/src/main/java/WGPUBuild.java`; it configures `BuilderTool.build(...)`, IDL parsing (`jWebGPU.idl`), custom C++ glue (`src/main/cpp/custom/jWebGPU.cpp/.h`), and platform targets.
- Service boundary: Java API shape comes from `webgpu-base` + `jWebGPU.idl`; native behavior lives in custom C++; platform packaging is in each target module `build.gradle.kts`.
- Runtime backend selection happens in `webgpu-base/.../JWebGPULoader.java`: DAWN vs WGPU, with non-Windows forced to WGPU.

## Source Of Truth (Edit vs Generated)
- Edit `webgpu/webgpu-base/src/main/java/**` and `webgpu/webgpu-build/src/main/cpp/**`.
- Do not hand-edit generated Java in:
  - `webgpu/webgpu-core/src/main/java/**`
  - `webgpu/webgpu-desktop-ffm/src/main/java/**`
  - `webgpu/webgpu-teavm/src/main/java/gen/**`
- Generated files include a `This file was generated by JParser` header (example: `webgpu-core/.../JWebGPULoader.java`).

## Project-Specific Conventions
- Native bridge blocks are embedded in `webgpu-base` Java using comment directives like `/*[-JNI;-NATIVE]*/`, `/*[-FFM;-NATIVE]*/`, `/*[-TEAVM;-REPLACE]*/` (example: `webgpu-base/.../WGPUBuffer.java`).
- IDL naming and binding metadata live in `webgpu-build/src/main/cpp/jWebGPU.idl` (e.g., `[Ref]`, `[BindTo]`, `[-NEW_OBJECT]`, `[-SKIP]`).
- Memory/allocation pattern: many wrappers expose static `Obtain(...)` for reuse; avoid replacing with per-frame `new` unless isolation is required (see README and IDL vector interfaces).
- TeaVM output is packaged as JS/WASM from `webgpu-build/build/c++/libs/emscripten` into `webgpu-teavm` jar.

## Integration Points
- Local development against jParser/TeaVM can be enabled via commented `includeBuild(...)` dependency substitution in `settings.gradle.kts`.
- Publishing/versioning: `gradle.properties` + `buildSrc/src/main/kotlin/LibExt.kt` control version (`-SNAPSHOT` vs release); publishing flow is in `buildSrc/src/main/kotlin/publish.gradle.kts` (`publishSnapshot`, `publishRelease`, `publishTestRelease`).
- Native classifier jars are assembled in `webgpu-desktop-jni/build.gradle.kts` and `webgpu-desktop-ffm/build.gradle.kts`.

## Debugging Shortcuts
- If generation looks wrong, inspect in order: `jWebGPU.idl` -> `webgpu-base` method directives -> generated target module file.
- If runtime loading fails, check expected native paths under `native/wgpu` or `native/dawn` and loader logic in `JWebGPULoader`.
- If TeaVM app starts but bindings are missing, verify `demos/app/teavm` task `findAndCopyJavaScriptInClasspath` copied `.js/.wasm` to `build/dist/webapp/scripts`.
